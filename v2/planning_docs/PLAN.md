# IK & Animation System Rework

## Problem Summary

1. **Marker duplication** тАФ BikeSkin has contact-point markers (seat, handlebars, pegs) and CharacterSkin has IK target markers for the same points (ButtPosition, LeftHand, RightHand, LeftFoot, RightFoot). Positions don't perfectly align.
2. **No per-bike body poses** тАФ `default_pose` is saved to a shared `IK_anim_lib.res`. Each bike needs different chest/head/magnet positions (sportbike tuck vs cruiser upright) but there's no way to store that per-bike.
3. **Hacky steering hand tracking** тАФ `_update_hand_ik_from_steering()` manually computes hand positions by rotating offsets around the steering pivot. Fragile math that doesn't generalize.
4. **Wheelie pivot sinks into ground** тАФ Pivot is at the bottom of the rear wheel. Past ~40 degrees, the contact point is actually the back of the tire, not the bottom. Bike clips through the ground.

## Design

### 1. Marker Consolidation тАФ Bike Owns Contact Points

**Bike owns** (in BikeSkin scene, saved to BikeSkinDefinition):
- `SeatMarker` тАФ rider's butt position (existing)
- `LeftHandleBarMarker` тАФ **reparented under RotationPoint** so it follows steering rotation automatically (existing marker, new parent)
- `LeftPegMarker` тАФ foot peg position (existing)
- Right side generated by mirroring X at runtime (existing pattern, no change)

**Character owns** (in CharacterSkin/IKTargets, body-shape markers):
- `ChestTarget`, `HeadTarget`
- `LeftArmMagnet`, `RightArmMagnet`
- `LeftLegMagnet`, `RightLegMagnet`

**Removed from CharacterSkin/IKTargets:**
- `ButtPosition` тАФ IKController reads `bike_skin.seat_marker` instead
- `LeftHand`, `RightHand` тАФ IKController reads bike's handlebar markers instead
- `LeftFoot`, `RightFoot` тАФ IKController reads bike's peg markers instead

**IKController wiring:** `PlayerEntity._init_ik()` passes bike marker node references to IKController at runtime via a new `set_bike_markers()` method. IKController stores them as vars (not `@export`s). The FABRIK solver doesn't care where the target nodes live in the scene tree тАФ it just needs Node3D references.

For the right-side markers, `_init_ik()` creates mirrored proxy Marker3D nodes at runtime (children of BikeSkin) with X negated, same as the current `set_ik_targets_for_bike()` pattern. These proxies are what IKController references for RightHand/RightFoot.

### 2. Per-Bike Body Pose on BikeSkinDefinition

New fields on `BikeSkinDefinition` resource under a new `@export_group("Rider Pose")`:

```
chest_position: Vector3
chest_rotation: Vector3
head_position: Vector3
head_rotation: Vector3
left_arm_magnet_position: Vector3
right_arm_magnet_position: Vector3
left_leg_magnet_position: Vector3
right_leg_magnet_position: Vector3
```

**Editor tool buttons** (on AnimationController, same workflow):
- **"Init IK from Bike"** тАФ positions IK targets from bike markers (existing), now also loads rider pose fields from BikeSkinDefinition to set chest/head/magnets
- **"Save Default Pose"** тАФ writes the current character-owned marker transforms (chest, head, magnets) back to the `BikeSkinDefinition` resource via `ResourceSaver`. No longer writes to `IK_anim_lib.res`.
- **"Reset to Default Pose"** тАФ reads rider pose fields from BikeSkinDefinition and applies them to the character markers. No longer seeks an animation.

**Runtime init:** `AnimationController.initialize()` reads rider pose fields from `BikeSkinDefinition` and sets the character-owned markers, then captures `_base_*` positions. Replaces the current `IK_anim_lib/default_pose` animation seek.

### 3. Steering тАФ Hands Follow Handlebars Automatically

`LeftHandleBarMarker` is reparented under `RotationPoint` in the BikeSkin scene. When `bike_skin.rotate_steering()` rotates `RotationPoint`, the handlebar marker (and thus the hand IK target) moves with it. The right-side proxy mirror also follows because it copies from the left marker's global transform with negated X.

**Deleted code:**
- `AnimationController._update_hand_ik_from_steering()` тАФ entire method removed
- `AnimationController._base_left_hand_pos` / `_base_right_hand_pos` тАФ no longer needed
- The call to `_update_hand_ik_from_steering(ik_ctrl, blend)` in `_update_procedural_animation()`

Hand positions are now fully driven by the bike's steering node. No lerping or math needed.

### 4. Wheelie Pivot тАФ Lerp Along Tire Contact Arc

**New markers on BikeSkin** (saved to BikeSkinDefinition):
- `RearWheelBackMarker` тАФ position at the back of the rear tire
- `FrontWheelFrontMarker` тАФ position at the front of the front tire

**New fields on BikeSkinDefinition:**
```
rear_wheel_back_position: Vector3
front_wheel_front_position: Vector3
```

**Pivot calculation** in `_update_procedural_animation()`:

```gdscript
# Wheelie: lerp from bottom-of-wheel to back-of-wheel based on pitch
var pitch_ratio = clamp(abs(visual_root.rotation.x) / (PI / 2.0), 0.0, 1.0)
if visual_root.rotation.x < 0:
    # Wheelie тАФ rear wheel
    pivot = bd.rear_wheel_ground_position.lerp(bd.rear_wheel_back_position, pitch_ratio)
else:
    # Stoppie тАФ front wheel
    pivot = bd.front_wheel_ground_position.lerp(bd.front_wheel_front_position, pitch_ratio)
```

This traces the tire's circular contact arc. At 0 degrees, pivot is bottom of wheel. At 90 degrees, pivot is back (or front) of wheel. The lerp approximates the arc well enough for visual purposes.

### 5. What Stays the Same

- **Ragdoll system** тАФ completely untouched
- **IK_anim_lib.res** тАФ still used for trick/polish IK animations (e.g. `wheelie_arm_drag`), just no longer stores `default_pose`
- **CharacterSkin.set_ik_targets_for_bike()** тАФ removed (replaced by the new init flow)
- **Procedural animation** for lean, chest rotation, butt shift тАФ stays the same, just reads from bike markers for butt position instead of character markers
- **BikeSkin marker save/load buttons** тАФ still save marker positions to BikeSkinDefinition
